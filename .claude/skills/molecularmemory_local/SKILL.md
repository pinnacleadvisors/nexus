# /molecularmemory_local — Dense, linked, tag-free memory

## Why this exists
Tags in Obsidian-style systems degrade into a messy cloud. Every important **entity** (person, company, concept, decision) should instead get its **own note**, linked from other notes with `[[wikilinks]]`. Claude can then traverse the graph the same way a human would in Obsidian's graph view.

Inspired by the atomic-note-taking approach (Karpathy-adjacent, but without the tag sprawl):
- **Atomic note** = one fact per file (e.g. *"Mitochondria produce ATP via oxidative phosphorylation"*)
- **Molecule note** = combines atomic notes into a higher-level concept (e.g. *"Cell biology"* MOC)
- **Entity note** = a stable "thing" (person / company / project / concept)
- **MOC** (Map of Content) = a homepage for a topic, linking all related atomic + entity notes

## Where the notes live

```
memory/molecular/
├── INDEX.md             ← generated homepage — MOCs/entities/atoms/sources/synthesis counts
├── log.md               ← append-only chronological record of ingest / synthesis / lint events
├── atoms/<slug>.md      ← atomic notes (one fact each)
├── entities/<slug>.md   ← people, companies, concepts, projects
├── mocs/<slug>.md       ← Maps of Content (topic hubs)
├── sources/<slug>.md    ← page-per-ingested-document (URL/PDF/transcript summaries)
├── synthesis/<slug>.md  ← filed Q&A or multi-source essays
└── .graph.json          ← cached adjacency list (regenerated by `graph` cmd)
```

`memory/` already exists for platform docs; molecular memory lives in a subfolder so they don't collide.

## Note format

Every file starts with YAML frontmatter so `graph.mjs` can parse edges without guessing:

```markdown
---
type: atom | entity | moc
title: Human-readable title
id: slug-matching-filename
created: 2026-04-21
sources:
  - https://example.com/article      # where the fact came from
links:
  - "[[other-atom]]"                  # explicit outbound edges
  - "[[entity/Andrej-Karpathy]]"
---

One-sentence fact, or a short molecule description. Then prose.
Inline [[wikilinks]] also count as edges.
```

## Commands

All exit 0 and print JSON. Claude invokes them via Bash.

### `init`
Creates `memory/molecular/{atoms,entities,mocs}/` and seed `INDEX.md` if absent.

### `atom <title> --fact="..." [--source=URL] [--links=slug1,slug2]`
Creates a new atomic note. Slug auto-derived from title.

### `entity <type> <name> [--description="..."] [--aliases=a,b]`
`<type>` ∈ `person | company | concept | project`. Creates or updates an entity note.

### `moc <title> [--atoms=a,b] [--entities=x,y] [--description="..."]`
Creates a Map of Content linking the given atoms + entities.

### `source <title> --url=<url> [--summary="..." | --body=<path>] [--moc=slug] [--links=a,b]`
Creates a per-document page in `sources/`. The body comes either inline via `--summary`, or from a markdown file (e.g. the output of `firecrawl_local scrape`) via `--body=<path>`. If `--moc=` is set, the source is appended under a `## Sources` section in that MOC. An ingest line is added to `log.md` automatically.

### `synthesis <title> --body=<path> [--question="..."] [--moc=slug] [--links=a,b]`
Creates a synthesis page in `synthesis/` — used to file notable Q&A or cross-source essays. Same MOC + log integration as `source`.

### `ingest <url|path> --title="..." [--summary="..." | --body=<path>] [--moc=slug] [--links=a,b]`
Convenience wrapper around `source`. Captures the canonical pattern: scrape with `firecrawl_local` → save to `--body` → run `ingest` → extract atoms with `atom` and link them by editing the source page's "Atoms extracted" section.

### `log <op> "<title>" [--ref=path] [--url=...]`
Appends a single line to `memory/molecular/log.md` with date prefix. Used by `source`, `synthesis`, and `lint --write`; callable directly to record decisions.

### `lint [--stale-days=90] [--write]`
Reports knowledge-base health: dangling `[[wikilinks]]`, orphan atoms (no incoming links), atoms missing `sources:`, atoms older than the threshold, entities with no atoms. With `--write`, appends a summary to `log.md`.

### `extract <file-or-url>` *(Claude-driven)*
Reads the target (via Read or /firecrawl_local) and **lets Claude pick the atoms**. The skill only provides the storage primitives — Claude plans the atoms, then loops `atom` for each one. See "Workflow" below.

### `link <from-slug> <to-slug>`
Adds a reciprocal `[[...]]` edge between two notes.

### `graph [--json] [--dot]`
Scans all notes, builds adjacency list → writes `.graph.json`. With `--dot`, emits Graphviz. Prints `{nodes, edges, orphans, hubs}`.

### `query <text>`
Full-text search across titles + facts. Ranks by connectivity (a hit on a hub note outranks a hit on a leaf).

### `reindex`
Rebuilds `INDEX.md` from scratch — lists every MOC, entity count, atom count, orphan list.

## Claude's workflow for "ingest this article" (Karpathy-style)

1. `node .claude/skills/molecularmemory_local/cli.mjs init` (once)
2. Fetch the source: `/firecrawl_local scrape <url> --output=/tmp/<slug>.md` (or `Read <local-path>` for files already on disk)
3. `cli.mjs ingest <url> --title="<short title>" --body=/tmp/<slug>.md --moc=<topic-slug>`
   - Creates `sources/<slug>.md`, appends `## Sources` block to the MOC, and logs `## [date] ingest | …`
4. **Claude** reads the source page and plans atoms (one fact per bullet, no duplicates with existing notes)
5. For each planned atom: `cli.mjs atom "<title>" --fact="..." --source=sources/<slug>.md --links=<existing-slugs>`
   - Then edit the source page's "Atoms extracted" section to back-reference each atom via `[[atoms/<slug>]]`
6. For any new person/company/concept: `cli.mjs entity <type> "<name>"`
7. If a notable Q&A or cross-source insight emerged, file it: `cli.mjs synthesis "<title>" --body=/tmp/<answer>.md --question="..." --moc=<topic-slug>`
8. `cli.mjs graph && cli.mjs reindex` → refreshes `INDEX.md` and `.graph.json`
9. `cli.mjs lint --write` → flag orphans, dangling links, and atoms missing sources; appends summary to `log.md`
10. Commit: `git add memory/molecular && git commit -m "molecular: ingest <topic>"`

## Query flow (saves tokens vs re-reading source docs)

1. Start at `memory/molecular/INDEX.md` — it lists MOCs
2. Open the relevant MOC → it links the atoms and entities for that topic
3. Read only those atoms
4. If you need related concepts, follow the `[[wikilinks]]` on the atom

This is the same discipline as the existing `memory/INDEX.md` flow, extended to domain knowledge (not just platform docs).

## Design rules Claude must follow

- **No tags.** Ever. If you feel the urge to tag, make a new MOC or entity note instead.
- **One fact per atom.** If an atom has more than one sentence of core fact, split it.
- **Every person/company/concept gets an entity note**, even if it's one line. Stable anchors enable dense linking.
- **Links are mandatory.** Every new atom must link to at least one existing atom, entity, or MOC — or create them. Orphan notes are a smell.
- **Sources are mandatory.** An atom without a `sources:` entry is a claim, not a fact.
- **MOCs over folders.** Don't create deep directory trees; use MOCs to group.
- **Prefer editing over duplicating.** If an atom already exists, add a `[[backlink]]` rather than creating `atom-v2.md`.

## When to use this skill vs `memory/` platform docs

| Kind of knowledge                              | Location                            |
| ---------------------------------------------- | ----------------------------------- |
| Stack rules, env vars, file layout             | `memory/platform/*.md`              |
| Roadmap phase status                           | `memory/roadmap/*.md`               |
| Facts about the *domain* (users, competitors)  | `memory/molecular/atoms/`           |
| People / companies / external tools            | `memory/molecular/entities/`        |
| "What do I know about topic X?"                | `memory/molecular/mocs/<topic>.md`  |
| Session-durable decisions                      | `docs/adr/` (still the right place) |

## Limitations
- Storage is markdown files — no server, no embeddings, no vector search. Full-text only.
- Ranking is connectivity-weighted term matching; it won't find semantic neighbours the way an embedding store would.
- Dense linking requires discipline; the skill can't force it — Claude must follow the rules above.
